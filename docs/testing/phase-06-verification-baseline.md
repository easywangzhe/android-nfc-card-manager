# Phase 06 验证基线

## 目标

本基线用于把 Phase 1-5 已落地的权限、真实性、结果来源、推荐下一步与审计影响范围收口为同一套验证语言。

- 自动化回归与真机检查必须复用本文件里的验证对象、观测信号和通过口径
- 不允许测试直接依赖页面 copy 的偶然细节；优先依赖稳定 test tags、结构块和既有语义标签
- 解锁仍然属于 **Demo 流程**，任何成功态都不能被描述成“真实已解锁”

## 统一术语

| 术语 | 含义 | 典型来源 |
| --- | --- | --- |
| 真实性 | 当前能力或结果属于真实设备结果、Demo 流程、推断结果或待补充 | 能力说明、结果页标签、审计详情 |
| 结果来源 | 高风险流程结果来自已确认执行、执行失败、结果未验证或仅演示 | 锁卡/解锁结果 guidance |
| 推荐下一步 | 当前最安全、最明确的后续动作，不等于“唯一可执行操作” | 读卡/写卡/格式化 guidance |
| 影响范围 | 当前记录或页面说明主要影响安全性、可追责性或本地便利性 | 审计、模板、设置等辅助页 |
| 路由守卫 | 角色无权访问目标页面时，页面渲染拒绝态而不是继续保留原始操作页 | AppNavGraph + PermissionDeniedScreen |

## 验证对象总览

| 验证对象 | 核心问题 | 自动化优先级 | 真机补充必要性 |
| --- | --- | --- | --- |
| 登录/首页权限 | 登录后是否只展示当前角色可执行入口 | 最高 | 中 |
| 高风险真实性 | 锁卡/解锁页是否持续表达真实支持边界与 Demo-only 边界 | 最高 | 高 |
| 路由守卫 | 进入页面后角色变化时，是否会回到拒绝态 | 最高 | 中 |
| 高频结果页 | 读卡/写卡/格式化结果是否保留“发生了什么 / 为什么 / 推荐下一步”结构 | 高 | 高 |
| 审计语义 | 审计列表/详情能否读出角色、阶段、真实性、影响范围 | 高 | 中 |

---

## 1. 登录/首页权限

### 验证目标

确认登录页、首页三段式分组与首页入口可见性稳定复用角色权限，不因 UI 重构重新暴露越权入口。

### 观测信号

- 登录用户名、密码、登录按钮具备稳定 test tags
- 首页存在主任务 / 高风险操作 / 管理工具三个 section 观测节点
- 不同角色登录后，入口可见性与 `buildHomeSections(role)` 一致
- 审计员首页不出现读卡、写卡、锁卡、解锁入口
- 模板入口仅系统管理员可见

### 通过口径

#### 自动化通过

- `operator`：可见主任务、管理工具；不可见模板；不可见高风险 section
- `supervisor`：可见主任务、高风险操作、管理工具；不可见模板
- `admin`：可见主任务、高风险操作、管理工具；可见模板
- `auditor`：仅保留管理工具；日志与设置可见；主任务与高风险入口均不可见

#### 人工/真机补充通过

- 真机上登录、切换角色后首页不会短暂闪出旧角色入口
- 底部导航与首页入口不会出现“点进去才发现无权限”的主入口错觉

### 自动化覆盖方式

- 使用 instrumentation 登录本地演示账号
- 通过 `AppTestTags.LOGIN_*` 与 `AppTestTags.HOME_*` 定位输入框、section 和入口按钮
- 断言关键 section 与入口显隐，不依赖页面描述文案全文匹配

### 真机补充方式

- 真实设备上依次登录四个角色并观察首页首屏
- 切换角色后立即检查首页和底部导航是否同步变化

## 2. 高风险真实性

### 验证目标

确认锁卡、解锁页始终把“真实支持 / 未验证 / Demo 流程”边界放在显著位置，且不把解锁演示成功渲染为真实成功。

### 观测信号

- 锁卡页存在高风险摘要块、当前支持方式与真实性块、结果来源块
- 解锁页存在解锁能力边界块、当前状态与真实性块、结果来源块
- 解锁成功演示后仍显示“仅演示 / Demo 流程 / DEMO_ONLY”相关语义
- 锁卡/解锁结果 guidance 使用现有结果来源分类：`CONFIRMED_EXECUTED` / `FAILED` / `UNVERIFIED` / `DEMO_ONLY`

### 通过口径

#### 自动化通过

- 锁卡页进入后可稳定观测到“高风险操作”标签和真实性块
- 解锁页进入后可稳定观测到真实性块与 Demo-only 说明
- 点击“模拟解锁成功（仅演示）”后，结果区仍显示 Demo-only 结果来源，不出现等价于“真实已解除写保护”的表述

#### 人工/真机补充通过

- 真机贴卡时，处理中保护存在，用户无法误触返回或重复触发
- 演示成功与真实失败/未验证场景在视觉上可明确区分

### 自动化覆盖方式

- 使用 `AppTestTags.LOCK_*`、`AppTestTags.UNLOCK_*`
- 优先验证结构块、真实性标签、结果来源标签与演示按钮触发后的状态
- 不尝试自动控制真实 NFC 硬件

### 真机补充方式

- 在真实设备上验证“处理中请勿离开/重复贴卡”等风险保护
- 用至少一张不可解锁或永久只读卡确认 UI 不会误报解锁成功

## 3. 路由守卫

### 验证目标

确认用户即使已经进入页面，只要当前角色失去权限，也会立即回到拒绝态，而不是继续停留在可操作页面。

### 观测信号

- `PermissionDeniedScreen` 根节点与说明文案具备稳定 test tags
- 进入锁卡/解锁等受保护页面后切换角色，页面树会切回拒绝态
- 拒绝态说明文案与 `SecurityManager.accessDeniedMessage(...)` 对齐

### 通过口径

#### 自动化通过

- 主管或管理员进入锁卡/解锁页后，切换为无权限角色会显示拒绝页
- 拒绝页出现时，原高风险页面关键块不再可见

#### 人工/真机补充通过

- 真机上切换角色或恢复会话后，返回栈不会把用户带回已无权限的执行页

### 自动化覆盖方式

- 通过首页合法进入锁卡/解锁页
- 在页面内使用角色切换路径或共享角色状态变更后断言 `AppTestTags.PERMISSION_DENIED_*`

### 真机补充方式

- 真实设备操作中，从设置或首页完成角色切换后重新进入/返回页面，观察守卫生效时机

## 4. 高频结果页

### 验证目标

确认读卡、写卡、格式化结果继续保留结构化 guidance，而不是退化成单一 success/fail 文案。

### 观测信号

- 读卡结果页存在 recommendation 区块、真实性标签和 CTA
- 写卡结果页存在“写入执行结果 / 回读校验结果 / 推荐下一步”三段式
- 格式化页结果区存在“发生了什么 / 为什么 / 推荐下一步”结构
- 演示成功场景继续保留真实性边界，不把仅演示写卡当成真实设备写入成功

### 通过口径

#### 自动化通过

- 读卡结果页 recommendation 区块和 CTA 可被稳定定位
- 写卡模拟成功后，三段式 guidance 全部可见
- 格式化失败场景下，结果区仍能读出发生了什么、为什么和推荐下一步

#### 人工/真机补充通过

- 真机真实贴卡结果不会挤掉 recommendation/why/next-step 结构
- 写后回读失败时，校验失败与写入失败可以被区分

### 自动化覆盖方式

- 使用 `ReadResultStore`、演示写卡按钮和现有格式化错误入口构造页面状态
- 通过 `AppTestTags.READ_*`、`WRITE_*`、`FORMAT_*` 断言结构块与 CTA 存在

### 真机补充方式

- 至少对真实读卡成功、写卡成功、写后未验证、格式化失败各执行一次回归
- 观察 recommendation 与真实性标签是否与实际卡片状态一致

## 5. 审计语义

### 验证目标

确认审计列表和详情页能把 UI 流程里的角色、阶段、真实性和影响范围完整暴露给用户，并与主流程语义一致。

### 观测信号

- 审计列表每条记录可读出操作、操作者、角色、阶段、真实性、说明
- 审计详情包含“谁执行了什么 / 结果来源与真实性 / 影响范围 / 说明与后续参考”四个分区
- 详情页 message 与主流程 guidance 中的真实性边界一致

### 通过口径

#### 自动化通过

- 至少一条读卡或写卡/解锁记录能在详情页读出 `who / stage / authenticity / impact / message`
- 解锁演示成功写入的审计真实性仍为 `Demo 流程`
- 影响范围展示不为空，且继续使用既有标签（如可追责性）

#### 人工/真机补充通过

- 真机完成操作后立即进入审计页，能看懂本次动作的来源边界与后续说明
- 历史记录和新记录混排时，不会出现角色、真实性或影响范围空白

### 自动化覆盖方式

- 使用现有本地日志写入链路触发读卡、写卡或解锁演示日志
- 通过 `AppTestTags.AUDIT_*` 断言列表项与详情分区

### 真机补充方式

- 每条真机回归用例完成后立即核对审计记录
- 核对字段至少包括：角色、阶段、真实性、影响范围、说明

## 自动化分层策略

1. **优先锁壳层边界**：登录、首页、路由守卫、高风险真实性
2. **再锁高频结果结构**：读卡/写卡/格式化 guidance
3. **最后锁审计一致性**：列表与详情字段闭环

自动化不覆盖真实贴卡硬件行为本身；这部分必须由真机矩阵补位。

## 真机回归执行要求

- 每条真机用例必须同时记录：设备/系统、卡型/锁定方式、期望 UI、期望审计、失败恢复动作
- 执行结束后必须勾选是否满足“真实性、结果来源、推荐下一步、影响范围”四类观测项
- 如果发现页面文案或结果来源分类与本基线不一致，应先修正文档/测试契约分歧，再继续扩测试

## 与共享测试契约的映射

- `AppTestTags.LOGIN_*`：登录/首页权限
- `AppTestTags.HOME_*`：首页 section 与入口可见性
- `AppTestTags.PERMISSION_DENIED_*`：路由守卫
- `AppTestTags.LOCK_*` / `UNLOCK_*`：高风险真实性与结果来源
- `AppTestTags.READ_*` / `WRITE_*` / `FORMAT_*`：高频结果页结构
- `AppTestTags.AUDIT_*`：审计列表与详情语义

如需新增 test tag，必须先回答两个问题：

1. 它对应哪个验证对象？
2. 它承载的是结构观测，还是现有语义标签的稳定定位？

若回答不清楚，则不应新增 tag。
