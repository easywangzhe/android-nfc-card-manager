# Phase 06 真机 NFC 回归矩阵

## 使用方式

本矩阵覆盖读卡、写卡、格式化、锁卡、解锁五条主流程。每条用例都必须记录：

- 设备 / 系统
- 卡型 / 锁定方式
- 前置条件
- 触发步骤
- 期望 UI 阶段 / 真实性 / 结果来源
- 期望审计字段
- 失败恢复动作

> 说明：下表中的设备型号可替换为团队现有真机，但必须保留 Android 版本与 NFC 开启状态记录。

## 统一核对字段

| 维度 | 必查项 |
| --- | --- |
| UI 阶段 | 等待 / 扫描中 / 处理中 / 成功 / 失败是否符合流程 |
| 真实性 | 是否准确显示真实设备结果 / Demo 流程 / 推断结果 |
| 结果来源 | 高风险流程是否准确显示已确认执行 / 执行失败 / 结果未验证 / 仅演示 |
| 推荐下一步 | 是否给出当前最安全动作 |
| 审计 | 角色、阶段、真实性、影响范围、说明是否齐全 |

## 用例 1：读卡成功并给出后续建议

| 字段 | 内容 |
| --- | --- |
| 流程 | 读卡 |
| 设备 / 系统 | Pixel 7 / Android 15 |
| 卡型 / 锁定方式 | NDEF 可读写卡 / 未锁定 |
| 前置条件 | 以 `operator` 或 `supervisor` 登录；NFC 已开启；卡片可被系统识别 |
| 触发步骤 | 进入读卡流程 → 明确点击开始扫描 → 贴卡读取 → 进入读卡结果页 |
| 期望 UI 阶段 | 结果页显示读卡成功，保留“共享阶段”“读卡真实性”“下一步操作”结构 |
| 期望真实性 | 读卡真实性显示真实设备结果或与当前能力映射一致的真实性标签 |
| 期望结果来源 | 不适用高风险结果来源；但 recommendation/CTA 必须可见 |
| 期望审计字段 | 若本轮流程写入日志，详情需包含角色=操作员/主管、阶段=执行完成、真实性与说明不为空 |
| 失败恢复动作 | 若读卡失败或为空标签，返回重试；记录是否出现 recommendation 区块缺失 |

### 额外观察

- CTA 与 recommendation 不应消失
- 读卡成功后不应把后续写卡或锁卡真实性误写成“已确认执行”

## 用例 2：写卡演示成功与回读校验结构保留

| 字段 | 内容 |
| --- | --- |
| 流程 | 写卡 |
| 设备 / 系统 | Pixel 7 / Android 15 |
| 卡型 / 锁定方式 | NDEF 卡 / 未锁定 |
| 前置条件 | 以 `operator`、`supervisor` 或 `admin` 登录；写入内容非空；必要时准备可写 NDEF 卡 |
| 触发步骤 | 进入写卡页 → 输入内容或应用模板 → 使用真实贴卡写入，或点击“模拟写卡成功（仅演示）” |
| 期望 UI 阶段 | 结果区保留“写入执行结果 / 回读校验结果 / 推荐下一步”三段式 |
| 期望真实性 | 若走演示按钮，必须保留“仅演示”边界；若走真实贴卡，真实性应与实际能力匹配 |
| 期望结果来源 | 非高风险流程，不显示高风险结果来源；但执行结果与回读校验必须分离 |
| 期望审计字段 | 写卡日志详情应能读出角色、阶段、真实性、影响范围=可追责性、说明与校验信息 |
| 失败恢复动作 | 若回读校验失败，记录是否仍保留三段式；执行“重新扫描 / 重试写卡”并再次验证 |

### 额外观察

- `VERIFY_FAILED` 不能被 UI 降级成普通写入失败
- 演示写卡成功不能被用户误解为真实写入已完成

## 用例 3：格式化失败仍保留诊断与推荐下一步

| 字段 | 内容 |
| --- | --- |
| 流程 | 格式化 |
| 设备 / 系统 | Pixel 7 / Android 15 |
| 卡型 / 锁定方式 | 非 NDEF 或不可格式化卡 / 可能为只读或不支持 |
| 前置条件 | 以具备格式化权限的角色登录；准备一张不支持格式化或故意触发失败的卡 |
| 触发步骤 | 进入格式化页 → 点击开始格式化 → 贴卡触发失败 |
| 期望 UI 阶段 | 结果区仍包含“发生了什么”“为什么”“推荐下一步/建议 CTA” |
| 期望真实性 | 真实性块持续可见，不因失败而丢失 |
| 期望结果来源 | 非高风险流程，不显示高风险结果来源；失败原因必须可读 |
| 期望审计字段 | 若形成日志，详情页需可读出阶段=执行失败、真实性不为空、影响范围不为空、message 说明失败原因 |
| 失败恢复动作 | 退出页面前记录失败原因；如卡片本身不支持，建议返回读卡确认卡片能力 |

### 额外观察

- 失败态不能只剩“格式化失败”四个字
- recommendation 应指向更安全的后续动作，而不是诱导重复高风险尝试

## 用例 4：锁卡真实/未验证结果来源表达

| 字段 | 内容 |
| --- | --- |
| 流程 | 锁卡 |
| 设备 / 系统 | Pixel 7 / Android 15 |
| 卡型 / 锁定方式 | 支持密码保护或永久只读方案的卡 |
| 前置条件 | 以 `supervisor` 或 `admin` 登录；确认风险勾选与确认词可完成 |
| 触发步骤 | 进入锁卡页 → 观察风险摘要与真实性块 → 完成前置确认 → 真实贴卡或点击“模拟锁卡成功（仅演示）” |
| 期望 UI 阶段 | 处理中期间返回被禁用；结果页显示锁卡结果 guidance |
| 期望真实性 | 当前支持方式与真实性块持续存在；演示流程保持仅演示边界 |
| 期望结果来源 | 结果来源必须来自 `已确认执行 / 执行失败 / 结果未验证 / 仅演示` 之一 |
| 期望审计字段 | 详情页需可读出角色、阶段、真实性、影响范围=安全性或可追责性、message 与锁定方式信息 |
| 失败恢复动作 | 若 15 秒内未贴卡或卡片不支持锁卡，应出现失败说明与恢复建议；确认 ReaderMode 已释放 |

### 额外观察

- 处理中时不能误触返回或重复点确认锁卡
- 敏感字段展示应符合角色遮罩规则

## 用例 5：解锁 Demo-only 边界不漂移

| 字段 | 内容 |
| --- | --- |
| 流程 | 解锁 |
| 设备 / 系统 | Pixel 7 / Android 15 |
| 卡型 / 锁定方式 | 密码保护型演示卡，或任一可进入演示流程的卡 |
| 前置条件 | 以 `supervisor` 或 `admin` 登录；准备解锁理由和本地演示凭据 |
| 触发步骤 | 进入解锁页 → 核对“解锁能力边界”与真实性块 → 点击“模拟解锁成功（仅演示）”或执行真实识别后进入结果页 |
| 期望 UI 阶段 | 页面持续展示受控恢复语义；结果页有 guidance 与敏感字段区 |
| 期望真实性 | 成功态仍明确标记为 Demo 流程 / 仅演示，不得出现“真实已解锁”暗示 |
| 期望结果来源 | 成功演示必须显示 `仅演示 / DEMO_ONLY`；失败则显示 `执行失败`；未验证则显示 `结果未验证` |
| 期望审计字段 | 审计详情真实性必须为 Demo 流程；角色、阶段、影响范围、message 与 UI 边界一致 |
| 失败恢复动作 | 若真实卡片不支持解锁，应回到失败说明并提示更安全动作；不得继续停留在“成功”视觉状态 |

### 额外观察

- 解锁仍为流程演示，不应在任何 closeout 或截图中表述为“产品已支持通用解锁”
- 进入页面后切换为 `auditor` 或 `operator`，必须被路由守卫切回拒绝态

## 审计核对清单

每条真机用例完成后，立即打开审计列表与详情，至少检查以下字段：

| 字段 | 期望 |
| --- | --- |
| 谁执行了什么 | 执行人 + 角色 + 发生了什么 均可读 |
| 当前阶段 | 已发起 / 执行中 / 执行完成 / 执行失败 与实际结果一致 |
| 真实性 | 与当前流程真实边界一致，尤其是解锁 Demo-only |
| 影响范围 | 不为空，通常为安全性或可追责性 |
| 说明与后续参考 | 能解释当前结果来源与恢复建议 |
