# Android NFC 卡片管理 App 技术方案文档（V1）

## 1. 文档目标
本文档用于明确 Android NFC 卡片管理 App 的技术实现方案，覆盖整体架构、模块划分、关键流程、数据存储、安全设计、兼容性策略、异常处理与开发建议，作为研发实施依据。

## 2. 建设目标

### 2.1 目标
- 支持指定卡型的 NFC 读卡、写卡、锁卡、解锁
- 支持模板化写卡与操作日志留痕
- 支持高风险操作权限控制与审计
- 支持后续扩展更多卡型与后台管理能力

### 2.2 设计原则
- 卡型能力边界清晰，不做“通用支持所有卡”承诺
- 读写流程标准化，高风险流程强约束
- 核心 NFC 能力、业务规则、UI 状态分层解耦
- 兼顾 V1 快速落地和 V2 扩展性

## 3. 技术选型

### 3.1 客户端
- 语言：Kotlin
- 最低建议系统版本：Android 8.0+
- UI：Jetpack Compose
- 架构：MVVM + Clean Architecture
- 并发：Kotlin Coroutines + Flow
- 依赖注入：Hilt
- 本地存储：Room
- 序列化：kotlinx.serialization 或 Gson/Moshi
- 日志：Napier/Timber 或 Android Log + 本地审计封装

### 3.2 Android NFC 能力
- `NfcAdapter`
- `enableReaderMode()`
- `Tag`
- `TagTechnology`
  - `Ndef`
  - `NfcA`
  - `IsoDep`
  - `MifareUltralight`
  - `MifareClassic`（按设备能力启用）
  - `NdefFormatable`

### 3.3 服务端（可选，建议预留）
V1 可先本地运行，若有企业化要求，建议预留服务端接口能力：
- 用户/权限管理
- 模板管理与下发
- 审计日志上报
- 高风险操作审批

## 5. 核心模块设计

### 5.1 NFC 核心模块（core/nfc）
职责：统一封装 Android NFC 能力，向业务层屏蔽卡型差异。

#### 子模块
- `NfcSessionManager`：ReaderMode 生命周期控制
- `TagDispatcher`：根据 `Tag.techList` 分发卡片处理器
- `CardTechHandler`：各卡型技术处理器接口
- `NdefHandler`
- `MifareClassicHandler`
- `MifareUltralightHandler`
- `IsoDepHandler`
- `CardCapabilityResolver`：解析卡型支持能力（读/写/锁/解锁）

#### 统一输出模型
- `CardInfo`
- `CardCapability`
- `ReadResult`
- `WriteResult`
- `LockResult`
- `UnlockResult`

### 5.2 业务规则模块（domain）
职责：封装业务流程，不直接依赖 UI。

#### 主要规则
- 写卡前必须做模板校验
- 锁卡前必须判断卡型能力与是否可逆
- 解锁前必须校验权限、凭据和卡状态
- 高风险操作必须写审计日志

### 5.3 模板模块（feature/template + data/template）
职责：管理写卡模板。

能力：
- 模板新增/编辑/删除/停用
- 模板版本管理
- 模板字段校验
- 模板与卡型兼容性校验

### 5.4 审计模块（feature/audit + core/security）
职责：记录关键操作行为。

记录范围：
- 读卡
- 写卡
- 锁卡
- 解锁
- 模板管理
- 权限拒绝与异常行为

### 5.5 权限安全模块（core/security）
职责：控制页面访问、按钮显示、操作权限、敏感数据脱敏。

能力：
- 角色权限判断
- Scope 控制（本人/团队/全局）
- 敏感字段脱敏
- 高风险操作二次认证

## 6. 卡型能力抽象设计

### 6.1 抽象目标
不同 NFC 卡/标签能力差异大，技术上不应在业务代码中直接写死某一类卡逻辑，需抽象统一能力层。

### 6.2 建议能力模型

```text
CardCapability
- canRead: Boolean
- canWrite: Boolean
- canLock: Boolean
- lockMode: NONE | READ_ONLY_PERMANENT | PASSWORD_PROTECTED
- canUnlock: Boolean
- requiresAuthForWrite: Boolean
- supportsNdef: Boolean
- techType: NDEF | MIFARE_CLASSIC | ULTRALIGHT | ISO_DEP | UNKNOWN
```

### 6.3 V1 支持策略
- 标准 NDEF 标签：优先支持
- Mifare Ultralight / NTAG：按能力支持读写和部分锁定
- Mifare Classic：仅对已验证机型开放
- IsoDep：V1 只做识别预留，不做复杂行业卡通用实现

## 7. 关键业务流程技术实现

### 7.1 读卡流程
1. 页面进入扫描态
2. `NfcSessionManager` 开启 ReaderMode
3. 获取 `Tag`
4. `TagDispatcher` 识别卡型并选择处理器
5. 处理器读取基础信息与数据区
6. `CardCapabilityResolver` 输出能力信息
7. 结果返回 ViewModel
8. ViewModel 保存审计日志并更新 UI

### 7.2 写卡流程
1. 用户选择模板或输入数据
2. 执行字段校验与卡型兼容性校验
3. 扫描目标卡
4. 识别是否可写
5. 调用对应 Handler 写入
6. 写后立即回读校验
7. 写入结果入库并记录审计日志

### 7.3 锁卡流程
1. 扫描识别卡片
2. 判断是否支持锁卡
3. 读取当前锁定状态
4. 校验操作者权限
5. 执行二次认证
6. 调用对应 Handler 执行锁卡
7. 重新读取状态确认成功
8. 记录高风险审计日志

### 7.4 解锁流程
1. 扫描识别卡片
2. 判断卡片是否支持解锁
3. 校验卡片当前是否处于可解锁状态
4. 校验权限与输入凭据
5. 调用对应 Handler 执行解锁
6. 回读校验可写状态
7. 记录审计日志

## 11. 安全方案

### 11.1 密钥与凭据管理
- 不在日志中记录明文密码/密钥
- 本地仅保存必要的加密态短期凭据
- 敏感数据使用 Android Keystore 保护

### 11.2 高风险操作安全
- 锁卡、解锁、模板删除必须二次确认
- 必须记录操作者身份、时间、结果、设备信息
- 可选接入生物识别/PIN 作为二次认证

### 11.3 数据脱敏
- UID 默认掩码
- 审计导出默认脱敏
- 仅管理员可查看更完整字段，且需留痕

## 12. 异常处理方案

### 12.1 NFC 相关异常
- NFC 未开启
- 设备无 NFC 硬件
- 卡片移开导致中断
- 多卡干扰
- 不支持的卡型

### 12.2 业务异常
- 模板与卡型不兼容
- 卡片已锁不可写
- 凭据错误
- 权限不足
- 高风险审批未通过

### 12.3 系统异常
- 本地数据库异常
- 接口超时
- 审计写入失败

### 12.4 异常处理原则
- 用户可恢复异常：给重试入口
- 不可恢复异常：给明确说明和下一步建议
- 所有关键失败都要记录错误码

## 13. 兼容性策略

### 13.1 设备策略
- V1 建立已验证设备白名单
- Mifare Classic 只在验证通过机型开放
- 其他不稳定能力采用动态探测 + 灰度开放

### 13.2 卡型策略
- V1 明确支持卡型清单
- 不在清单内的卡统一展示“不支持”
- 锁卡/解锁能力必须按卡型动态判断
