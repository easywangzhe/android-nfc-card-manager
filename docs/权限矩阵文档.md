# Android NFC 卡片管理 App 权限矩阵文档（V1）

## 1. 文档目的
明确系统中各类角色可执行的页面访问、功能操作、审批权限与数据可见范围，作为后续账号体系、接口鉴权、前端按钮控制和审计规则的依据。

## 2. 角色定义

### 2.1 普通操作员
面向一线使用人员，负责读卡、写卡、查看本人操作结果。

### 2.2 主管/审核人
负责高风险操作审批，可查看团队范围日志，可对解锁和锁卡进行审批确认。

### 2.3 系统管理员
负责模板维护、权限策略配置、高风险操作执行、全局日志查询。

### 2.4 审计员
负责查看全量审计日志与导出，不参与日常卡片操作。

## 3. 权限设计原则
1. 默认最小权限
2. 高风险操作必须与普通写卡分离
3. 审批权与执行权可分离
4. 所有高风险操作必须留痕
5. 数据查看遵循按角色、按范围、按敏感级别控制

## 6. 关键操作审批规则

### 6.1 读卡
- 无需审批
- 记录操作日志

### 6.2 写卡
- 默认无需审批
- 若写入模板属于高敏模板，可配置为“主管审批后可写”

### 6.3 锁卡
- 必须为高风险操作
- 默认流程：主管发起/审批，管理员执行
- 若企业策略较简化，也可配置为“管理员直接执行”

### 6.4 解锁
- 必须为高风险操作
- 默认流程：主管审批，管理员执行
- 必须填写解锁理由

### 6.5 模板删除
- 必须二次确认
- 建议仅做软删除
- 必须记录删除原因

## 7. 数据可见范围

### 7.1 卡片数据
- 普通操作员：仅查看本人操作涉及的数据
- 主管：查看团队范围内数据
- 管理员：查看全部数据
- 审计员：查看全部，但默认脱敏

### 7.2 敏感字段
以下字段应默认脱敏：
- 卡 UID
- 密钥/密码
- 审批码
- 设备唯一标识

### 7.3 明文查看策略
- 仅系统管理员在特定场景可查看完整数据
- 明文查看行为必须记录审计日志

## 8. 前端交互控制规则
1. 无权限按钮直接隐藏或置灰
2. 可申请但不可直接执行的操作显示“申请权限/提交审批”
3. 高风险操作按钮必须使用警示样式
4. 无权进入的页面在路由层直接拦截
5. 接口鉴权失败时返回统一错误并记录安全日志

## 9. 后端鉴权建议
建议采用 RBAC：
- Role：角色
- Permission：具体操作权限
- Scope：数据范围（本人/团队/全局）

建议权限编码示例：
- `card.read`
- `card.write`
- `card.lock.request`
- `card.lock.approve`
- `card.lock.execute`
- `card.unlock.request`
- `card.unlock.approve`
- `card.unlock.execute`
- `template.read`
- `template.manage`
- `audit.read.self`
- `audit.read.team`
- `audit.read.all`
- `audit.export`
- `system.admin`

## 10. 推荐落地策略（V1）

### 方案 A：简化版
- 普通操作员：读卡、写卡、本人日志
- 管理员：锁卡、解锁、模板、全量日志

适合早期快速上线。

### 方案 B：标准企业版
- 普通操作员：读卡、写卡
- 主管：锁卡/解锁审批、团队日志
- 管理员：高风险执行、模板管理、系统配置
- 审计员：全量日志只读

适合正式商用。
