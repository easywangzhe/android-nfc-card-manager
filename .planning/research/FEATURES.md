# Feature Landscape

**Domain:** Android NFC 卡片管理工具（现有能力上的功能优化 + UI 升级）  
**Researched:** 2026-03-31  
**Scope note:** 当前应用已具备登录、权限、读卡、写卡、锁卡、解锁、模板、审计、设置；本文件讨论的是**下一轮必须补齐的体验型/功能型要求**，不是新增一套完全不同的产品能力。  
**Overall confidence:** HIGH（对项目现状） / MEDIUM（对行业期望排序）

## Judgment Summary

这类 NFC 管理工具进入“优化期”后，用户真正期待的不是更多菜单，而是三件事：

1. **更清楚**：我现在在哪一步、系统读到了什么、接下来该做什么。
2. **更安全**：高风险动作不会误触、不会把 demo 能力包装成真实成功、不会留下不可追责的记录。
3. **更高效**：高频流程少走弯路，但不能以牺牲确认和可审计性为代价。

因此，本轮优先级应聚焦：**UI 清晰度、流程效率、风险控制、可审计性**。不要把资源投入到服务化、全新协议接入、花哨视觉、消费级功能上。

## Table Stakes

这些不是“锦上添花”，而是进入优化阶段后用户会默认应该被解决的问题；缺失会让现有能力继续显得不可信或难用。

| Feature | Why Expected | Complexity | 依赖 | 对现有系统侵入程度 | 主要触点 | Notes |
|---------|--------------|------------|------|------------------|----------|-------|
| 首页任务分层重组（主任务 / 高风险 / 管理入口） | 当前已有功能很多，用户首先期待入口清楚，而不是继续堆入口 | 中 | 现有权限矩阵、导航结构 | 中 | `feature/home/`, `navigation/` | 必须保留角色差异，不能因为“更整洁”而把高风险入口做成所有人可见可进 |
| 统一流程状态模型（准备中 / 等待贴卡 / 执行中 / 成功 / 失败 / 未验证 / 未实现） | 读写锁解页面已经存在，用户会默认不同页面的状态表达一致 | 中 | ViewModel 状态收口、统一结果模型 | 中 | `feature/read/`, `write/`, `lock/`, `unlock/`, `format/` | 尤其要显式区分 `成功`、`未验证`、`DEMO_ONLY/NOT_IMPLEMENTED` |
| 显式开始扫描与单次会话控制 | NFC 工具场景里，自动开始/重复触发会被视为危险设计，不是“便捷” | 中 | ReaderMode 生命周期治理、in-flight 锁 | 中 | 扫描/写卡/锁卡/解锁/格式化相关 Screen | 高频低风险页可“自动准备”，但高风险页不能“自动执行” |
| 写卡结果拆分为“写入结果”与“回读验证结果” | 现场用户真正关心的是“到底写上了没有，还是只是没验证到” | 中 | 写卡状态机、二次贴卡验证路径 | 中 | `feature/write/`, `core/nfc/NdefWriter.kt` | 必须避免把“验证失败”粗暴展示为“写入失败” |
| 高风险操作前置确认强化（锁卡/解锁/格式化） | 当前已有锁卡/解锁，用户会默认风险说明、条件确认、结果反馈都足够严肃 | 中 | 权限校验、统一风险组件、结果模型 | 中 | `feature/lock/`, `feature/unlock/`, `feature/format/` | 应包含风险摘要、前置条件、确认词/理由、执行后结果来源 |
| 能力真实性标记与 demo 骨架隔离 | 现有解锁/部分锁卡能力有骨架性质；继续假装“真实成功”会直接损害可信度 | 低-中 | 结果态建模、文案治理、审计字段补充 | 中 | `feature/unlock/`, `feature/lock/`, `core/nfc/`, `README` 对应文案 | 这是本轮必须立刻补的信任基线 |
| 审计上下文补齐（操作者、角色快照、阶段、真实性、脱敏卡标识） | 工具型产品在优化期，日志不能只“能看”，必须“能追责” | 中-高 | 会话信息接入、日志模型扩展 | 高 | `core/database/`, 各 `*ViewModel` | 当前 `operatorId` 恒为默认值，这一项直接影响产品可信度 |
| 敏感信息分级展示与关键页面防旁观 | 结果页、日志页、模板页已经存在，用户会默认敏感标识至少可控可脱敏 | 中 | 角色判断、详情页 UI 调整 | 中 | `feature/read/`, `audit/`, `write/`, `settings/` | UID、模板内容、凭据提示应按角色/动作分级展示 |
| 权限可见性、路由可达性、执行校验三层一致 | 既然已有登录与角色权限，用户会默认“看不见/进不去/不能执行”是一致的 | 中 | 安全层收口、导航守卫统一 | 中-高 | `core/security/`, `navigation/`, 首页入口 | 不能继续存在“首页切角色即可看到并执行高风险操作” |
| 模板 / 审计 / 设置页的信息层级整理 | 主流程优化后，辅助页若仍旧像原型拼装，会拉低整体可用性 | 低-中 | 统一组件、列表/详情布局规范 | 低-中 | `feature/template/`, `audit/`, `settings/` | 属于本轮应覆盖的收尾项，但优先级低于主流程与高风险流程 |

## Differentiators

这些有价值，但不应阻塞本轮。适合在主流程可信、风险边界清楚之后再做。

| Feature | Value Proposition | Complexity | 依赖 | 对现有系统侵入程度 | Notes |
|---------|-------------------|------------|------|------------------|-------|
| 审计导出 / 只读分享包 | 方便外部复核、线下归档、现场留痕 | 中 | 审计模型先补齐 | 中 | 先把日志“记对”，再谈“导出得漂亮” |
| 模板增强（收藏、最近使用、分类、版本备注） | 提升重复写卡效率，适合高频操作员 | 中 | 模板数据模型与列表交互优化 | 中 | 当前先解决模板页可读性与调用效率，再做治理增强 |
| 最近任务快捷入口 / 上次操作恢复 | 减少重复进入层级，提升班组作业效率 | 中 | 首页重组、会话恢复策略 | 中 | 只有在权限与风险提示不被绕开的前提下才值得做 |
| 平板/横屏双栏信息布局 | 适合设备管理台、日志详情、模板编辑并行查看 | 中 | 设计系统稳定、可能需要 adaptive 组件 | 中 | 如果目标设备以手机为主，可延后 |
| 卡型能力提示增强（“可尝试/已确认支持/不支持”） | 降低现场误判，帮助用户理解为什么某卡不能写/锁/解 | 中-高 | 更细的能力探测与错误映射 | 中 | 有价值，但应放在统一状态和结果模型之后 |
| 面向现场的失败恢复助手 | 如“重贴卡验证”“改用读卡确认”“检查 NFC 开关”分场景建议 | 低-中 | 错误模型统一 | 低-中 | 适合作为本轮后半段加分项，而不是前置项 |

## Anti-Features

本轮应明确排除；强行塞入只会分散资源，甚至把 UI 优化项目拖成失控重构。

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| 新增远程账号体系、后端服务化、在线同步 | `PROJECT.md` 已明确 offline-first，本轮核心问题也不是服务能力不足，而是现有流程不够清楚可信 | 先把本地权限边界、会话可信度、审计质量做好 |
| 接入真实通用解锁协议 / 大规模扩展新卡型协议 | 这是底层能力扩张，不是“功能优化 + UI 升级”；复杂度高且验证成本大 | 先把现有骨架能力显式标注为 demo / 未实现，避免误导 |
| 大规模架构重写（多模块、全量 Clean Architecture、全面替换状态框架） | 当前是 brownfield 优化，不应为了“更优雅”牺牲交付节奏与回归风险 | 先做状态模型统一、页面边界收口、局部 coordinator 化 |
| 批量自动写卡 / 自动连续执行高风险操作 | 工具场景里这会显著放大误操作与追责风险，不适合作为当前优化目标 | 先把单卡流程做稳、做清楚、做可审计 |
| 为了“更现代”加入大量动画、视觉特效、品牌化重皮肤 | 用户核心痛点不是“不够炫”，而是“不够清楚”“不够稳” | 用统一状态组件、风险语义和信息层级提升质感 |
| 消费级功能建议（社交、积分、分享战绩、个性化主题优先） | 与企业/工具型 NFC 管理场景无关，会稀释产品定位 | 只保留提升作业效率与风险控制的功能 |
| 把日志清理、缓存清理、会话清理继续混成一个入口 | 这是典型危险“便利功能”，会直接破坏可审计性 | 拆分为缓存清理与受控日志管理，两者权限和确认不同 |

## Feature Dependencies

```text
权限三层一致（入口/路由/执行）
  -> 首页任务分层重组

统一流程状态模型
  -> 读卡/写卡/锁卡/解锁/格式化页面统一改版
  -> 失败恢复助手

显式开始扫描 + 单次会话控制
  -> 写卡结果拆分
  -> 高风险操作前置确认强化

能力真实性标记
  -> 高风险结果页可信化
  -> 审计上下文补齐

审计上下文补齐
  -> 审计导出 / 只读分享包

统一组件与信息层级
  -> 模板 / 审计 / 设置页整理
  -> 平板/横屏双栏布局
```

## MVP Recommendation for This Round

优先做：

1. **能力真实性澄清 + 权限三层一致**  
   先堵住“能误进、能越权、会假成功”的信任缺口。

2. **统一流程状态模型 + 显式开始扫描/单次会话控制**  
   这是读写锁解 UI 升级的公共地基。

3. **高频流程页优化（读卡 → 写卡）**  
   先把最常用流程做清楚，再把模式复制到高风险页。

4. **高风险页优化（锁卡 / 解锁 / 格式化）**  
   聚焦风险说明、阶段提示、真实性标记和恢复路径。

5. **审计模型补齐后，再整理日志 / 模板 / 设置页**  
   不要先做“日志页美化”，再继续保留错误日志语义。

Deferred：

- **审计导出 / 分享包**：先把日志字段补对。
- **模板收藏 / 分类 / 版本**：先把模板页基础可用性和调用效率做好。
- **平板双栏 / adaptive 布局**：仅在目标设备明确需要时再做。
- **卡型能力提示增强**：等基础状态机与错误映射稳定后再做。

## Invasion Guidance

为了让下游 REQUIREMENTS 更容易拆解，建议按侵入程度理解本轮工作：

- **低侵入**：模板/日志/设置页布局整理、失败恢复文案、敏感字段默认脱敏
- **中侵入**：首页重组、统一状态组件、结果页语义统一、显式开始扫描
- **高侵入**：审计模型补齐、权限三层一致、能力真实性建模、ReaderMode 单次会话治理

结论：**本轮不是“纯 UI 美化”。真正必须覆盖的是那些同时影响可用性与可信度的体验型要求。**
